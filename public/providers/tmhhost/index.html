基础算力与内存读写跑分#

为了评估这台机器的底层算力，我首先使用 sysbench 对 CPU 与内存进行了常规压测。这直接关系到我们的服务在面对突发并发时的响应速度。下面是终端跑分时输出的核心结果行。

磁盘随机与顺序读写性能#

相比于纯粹的 CPU 算力，建站用户往往更需要关注磁盘的 IOPS 表现。这决定了数据库高频查询时是否会遇到严重的性能瓶颈。我通过 fio 工具分别测试了 4K 随机读写与 1M 连续读写的极限速度。

root@001:~# fio --name=randrw --rw=randrw --bs=4k --size=512M --numjobs=1 --runtime=60 --time_based --group_reporting
  read: IOPS=5088, BW=19.9MiB/s (20.8MB/s)(1193MiB/60001msec)
  write: IOPS=5085, BW=19.9MiB/s (20.8MB/s)(1192MiB/60001msec); 0 zone resets

root@001:~# fio --name=seqrw --rw=readwrite --bs=1M --size=512M --direct=1 --numjobs=1 --iodepth=1 --group_reporting
  read: IOPS=416, BW=417MiB/s (437MB/s)(242MiB/581msec)
  write: IOPS=464, BW=465MiB/s (487MB/s)(270MiB/581msec); 0 zone resets

从原始数据可以看出，4K 随机读写的 IOPS 稳定在 5000 左右。这个成绩说明硬盘底层进行了合理的限速配置，不会因为同宿主机邻居的突发读写导致你的服务卡顿。对于中小型 MySQL 数据库的日常运行来说，这个性能表现很实在。
